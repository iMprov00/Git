Давайте подробно разберем, что от вас хотят в этом задании, зачем это нужно, и как это сделать правильно.

## 📚 **Что такое fork workflow?**

Когда вы участвуете в open-source проектах, вы обычно:
1. **Не имеете прав** писать напрямую в оригинальный репозиторий
2. Делаете **fork** (копию) чужого репозитория к себе на GitHub
3. Работаете в своем форке, а потом предлагаете изменения автору

Для этого нужно настроить два удаленных репозитория (remote):
- **origin** — ваш форк (куда вы пушите свои изменения)
- **upstream** — оригинальный проект (откуда вы тянете обновления)

## 🎯 **Что требуется в задании:**

### Шаг 1: Создать новый репозиторий fork-practice
```bash
# Создаем папку
mkdir fork-practice

# Переходим в нее
cd fork-practice

# Инициализируем Git-репозиторий
git init
```
**Объяснение:** 
- `mkdir` — создает пустую папку
- `git init` — превращает обычную папку в Git-репозиторий (появляется скрытая папка `.git`)

### Шаг 2: Добавить remote origin (ваш форк)
```bash
git remote add origin git@github.com:yourname/project.git
```
**Объяснение:**
- `git remote add` — команда для добавления нового удаленного репозитория
- `origin` — традиционное имя для вашего собственного форка
- `git@github.com:yourname/project.git` — SSH-адрес вашего форка (замените `yourname` на свой логин)

Проверяем:
```bash
git remote -v
```
Должны увидеть:
```
origin  git@github.com:yourname/project.git (fetch)
origin  git@github.com:yourname/project.git (push)
```

### Шаг 3: Добавить remote upstream (оригинальный проект)
```bash
git remote add upstream git@github.com:original/project.git
```
**Объяснение:**
- `upstream` — традиционное имя для оригинального репозитория
- `git@github.com:original/project.git` — адрес проекта, который вы форкнули

Проверяем:
```bash
git remote -v
```
Теперь видим два remote:
```
origin    git@github.com:yourname/project.git (fetch)
origin    git@github.com:yourname/project.git (push)
upstream  git@github.com:original/project.git (fetch)
upstream  git@github.com:original/project.git (push)
```

### Шаг 4: Настроить, чтобы upstream использовался только для fetch (не для push)

**Проблема:** Сейчас `upstream` настроен и на fetch (скачивать), и на push (отправлять). Но в оригинальный проект вы пушить **не должны** (у вас нет прав, да и не нужно). Нужно убрать возможность push в upstream.

**Решение:** Установить для upstream пустой URL для push
```bash
git remote set-url --push upstream no_push
```
**Объяснение:**
- `git remote set-url --push` — меняет URL только для push операций
- `upstream` — имя remote, который меняем
- `no_push` — специальное значение, которое блокирует push (можно использовать любой несуществующий URL)

**Альтернативный вариант:**
```bash
git remote set-url --push upstream DISABLE
```
или
```bash
git remote set-url --push upstream ""
```

## ✅ **Финальная проверка:**

```bash
cd fork-practice && git remote -v | grep upstream | grep "(push)" | awk '{print $2}'
```

Эта команда должна вывести то, что вы указали для push URL (например, `no_push` или пустую строку).

## 📋 **Полный список команд для задания:**

```bash
# 1. Создаем репозиторий
mkdir fork-practice
cd fork-practice
git init

# 2. Добавляем origin (ваш форк)
git remote add origin git@github.com:yourname/project.git

# 3. Добавляем upstream (оригинал)
git remote add upstream git@github.com:original/project.git

# 4. Отключаем push для upstream
git remote set-url --push upstream no_push

# 5. Проверяем результат
git remote -v
```
